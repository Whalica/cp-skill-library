# 题目难度评分 Skill v4

## 1. 目标

本 Skill 用于在不直接依赖官方 rating 的前提下，对算法竞赛题目进行结构化难度评价，并同时输出：

1. **公共难度**
   - 题目在标准竞赛环境下，对普通竞赛群体的平均赛时 AC 难度。
   - 用于估算题目大致对应的 CF rating 档位。

2. **目标人群体感难度**
   - 指定水平与能力画像的选手，在赛时面对该题时的实际负担。
   - 允许目标人群在某些专题明显强于或弱于公共群体。

3. **赛时波动**
   - 描述不同选手、不同场次之间体感差异的离散程度。
   - 赛时波动不直接计入最终分，只影响估分区间与置信度。

4. **估分置信度**
   - 描述当前估分结论的可靠程度。
   - 受题面完整度、解法确定性、数据强弱、题型冷门程度等因素影响。

核心原则：

```text
公共难度 ≠ 目标人群体感难度
难度中心 ≠ 赛时波动
赛时波动不直接加分
目标人群不得重复校准
```

---

## 2. 标准竞赛环境

公共难度必须基于统一环境评价，否则不同题目的分数不可比较。

默认环境：

```yaml
standard_context:
  platform_style: "Codeforces 风格综合竞赛"
  contest_duration: "2 小时 15 分钟"
  language: "C++17"
  participant_profile: "持续参加 rated contest 的普通竞赛选手"
  editorial_access: false
  external_search: false
  local_template_allowed: true
  subtasks: false
  interactive: false
  statement_quality: "正常"
  sample_quality: "正常"
  pretest_strength: "正常"
  system_test_strength: "正常"
  time_limit_pressure: "正常"
  memory_limit_pressure: "正常"
```

若题目实际环境不同，必须在“环境修正”中说明，例如：

- 结果填空；
- 输出答案；
- 部分分；
- 交互；
- 通信；
- 开放数据；
- 极弱 pretest；
- 极强卡常；
- 非 C++ 主场；
- 允许长时间离线计算。


- 若用户未指定目标人群，使用默认配置。

## 4. 评分范围

所有基础维度均固定纳入，分数为：

```text
0.0 ~ 10.0
```

保留一位小数。

| 分数 | 含义 |
|---:|---|
| 0.0 | 该维度几乎没有负担 |
| 2.0 | 较低负担 |
| 4.0 | 中等偏低 |
| 5.0 | 中等 |
| 6.0 | 中等偏高 |
| 8.0 | 较高 |
| 10.0 | 极高 |

重要规则：

```text
某维度不存在时记 0.0，不允许“不纳入”。
```

这样可以避免“增加一个简单门槛反而使平均分下降”的数学问题。

# 5. 基础评价维度

基础维度共七项，权重固定，总和为 1.00。

| 维度 | 权重 |
|---|---:|
| 题意理解与形式化 | 0.10 |
| 前置知识门槛 | 0.15 |
| 核心观察与结论发现 | 0.20 |
| 算法设计与复杂度优化 | 0.20 |
| 正确性论证与反例排除 | 0.10 |
| 实现、调试与性能工程 | 0.15 |
| 错路诱导与解法路径宽度 | 0.10 |

公共难度与目标人群体感分别对这七项评分。

---

## 5.1 题意理解与形式化

评价从自然语言题面转化为明确数学对象、状态或操作模型的难度。

包括：

- 题面是否长、绕、翻译质量差；
- 操作时序是否复杂；
- 是否需要识别等价模型；
- 是否难以确定真正需要优化的量；
- 环、树、集合、偏序、博弈等抽象是否自然。

| 分数 | 特征 |
|---:|---|
| 0~2 | 题意直接，模型几乎显然 |
| 3~5 | 需要整理状态或做一层抽象 |
| 6~8 | 题意与算法对象差异较大，建模是主要门槛 |
| 9~10 | 题意高度复杂，形式化本身接近解题主体 |

边界：

- 不评价关键结论是否难发现；
- 不评价从模型到高效算法的优化过程；
- 只评价“题面到底在做什么，以及如何精确定义它”。

---

## 5.2 前置知识门槛

评价解决该题所需知识的稀有度、熟悉度和记忆要求。

需要区分：

```text
知识是否少见
```

与：

```text
掌握知识后，应用是否困难
```

前者属于本维度，后者属于算法设计或观察。

| 分数 | 特征 |
|---:|---|
| 0~2 | 基础语法、排序、前缀和、简单枚举 |
| 3~5 | 二分、基础图论、普通 DP、并查集、Fenwick、常见数论 |
| 6~8 | 线段树复杂维护、字符串算法、网络流、树上博弈、Burnside、冷门结论 |
| 9~10 | 极冷门专题、高级图论、高级代数或强依赖专门知识 |

注意：

- 常见模板代码长，不代表知识稀有度高。
- 公式极短但普通选手几乎没见过，也可能得高分。
- 存在合理无模板解时，应按最容易赛时到达的路径评分。

---

## 5.3 核心观察与结论发现

评价关键性质、构造、规律或突破口在赛时被自然发现的难度。

重点是发现路径，而不是结论表述看起来是否陌生。

必须进行“防后见之明检查”：

1. 不知道正解时，最自然的第一步是什么？
2. 普通选手为什么会观察这个量？
3. 普通随机小数据是否能暴露规律？
4. 是否必须先猜中观察对象，才能看出结论？
5. 反例是否容易构造？
6. 样例是否真实提供了方向，而非事后牵强解释？

| 分数 | 特征 |
|---:|---|
| 0~2 | 题面几乎直接给出，或自然推一步即可 |
| 3~5 | 手推、画图、枚举小数据后较自然 |
| 6~8 | 需要针对性推导，普通尝试不容易发现 |
| 9~10 | 单点结论极隐蔽，缺少可达发现路径 |

短代码不等于观察简单。

---

## 5.4 算法设计与复杂度优化

评价从正确性质或模型出发，设计出满足约束的算法有多难。

包括：

- 从暴力到目标复杂度的优化；
- 状态设计；
- 状态压缩；
- 数据结构中维护什么信息；
- 离线、分治、扫描线、摊还、增量维护；
- 去掉一个复杂度维度；
- 时间复杂度和空间复杂度的达成；
- 多个算法模块的组合。

| 分数 | 特征 |
|---:|---|
| 0~2 | 观察得到后直接实现即可 |
| 3~5 | 一到两层常规状态设计或复杂度优化 |
| 6~8 | 需要非显然状态、降维或多模块组合 |
| 9~10 | 算法结构深，复杂度达成路径极窄 |

该维度是 v4 新增的核心项目，用于避免“知道模板”和“会设计算法”被混为一谈。

---

## 5.5 正确性论证与反例排除

评价选手能否在赛时确认方案正确，而不仅仅是猜到一个看起来合理的做法。

包括：

- 贪心交换论证；
- 必要性与充分性；
- 单调性证明；
- 状态转移完整性；
- 构造是否覆盖全部情况；
- 是否存在隐蔽反例；
- 是否容易获得提交信心。

| 分数 | 特征 |
|---:|---|
| 0~2 | 正确性几乎显然 |
| 3~5 | 常规归纳、交换或不变量证明 |
| 6~8 | 猜法容易但证明困难，反例不易排除 |
| 9~10 | 结论高度反直觉，严格证明接近题目主体 |

注意：

- “能写暴力对拍”只能降低验证风险，不能替代数学证明难度。
- 某题可猜对但难以放心提交，本维度应提高。

---

## 5.6 实现、调试与性能工程

同时评价逻辑实现和工程约束。

建议内部拆分：

```text
逻辑实现与边界
性能与资源工程
```

考虑：

- 代码量；
- 状态数量；
- 分类讨论；
- 下标与边界；
- 溢出与取模；
- 多测清空；
- 递归栈；
- 内存布局；
- cache；
- 卡常；
- 语言差异；
- debug 成本；
- 构造输出验证；
- 交互协议。

| 分数 | 特征 |
|---:|---|
| 0~2 | 简单循环或少量判断 |
| 3~5 | 普通实现，有少量边界 |
| 6~8 | 长代码、状态多、性能压力明显 |
| 9~10 | 大模拟、复杂几何、交互、通信或极端工程题 |

强制规则：

- 实现分 `>= 7.0` 时，公共最终分通常不应低于 `5.5`。
- 实现分 `>= 8.0` 且边界密集时，公共最终分通常不应低于 `6.0`。
- 强选手可以降低体感，但不能把长实现风险直接降为 0。

---

## 5.7 错路诱导与解法路径宽度

评价题目是否容易把选手引向错误方向，以及正确路径是否过窄。

包括：

- 假贪心；
- 假单调性；
- 状态爆炸的错误 DP；
- 误用高级数据结构；
- 弱样例导致伪规律；
- 多个看似合理但错误的方向；
- 正解是否只有一条狭窄路径；
- 错误方向能否快速证伪。

| 分数 | 特征 |
|---:|---|
| 0~2 | 正确方向自然，错路少 |
| 3~5 | 有一些常见误区 |
| 6~8 | 错路多，排错成本高 |
| 9~10 | 正解反直觉且路径极窄 |

注意：

- 本维度不重复计算“核心观察本身有多难”。
- 只评价搜索空间、错误反馈和路径宽度。

---

# 6. 基础分计算

公共基础分：

\[
B_{\text{public}}=\sum_i w_i s_i
\]

目标人群基础体感：

\[
B_{\text{target}}=\sum_i w_i t_i
\]

其中：

- \(s_i\) 为公共维度分；
- \(t_i\) 为目标人群维度分；
- 所有维度固定纳入；
- 权重总和为 1。

禁止：

- 删除低分维度；
- 先对目标人群逐项降分，再通过 rating gap 二次降分；
- 因为结果“看起来不对”而随意调整某个项目。

---

# 7. 瓶颈与组合效应

简单平均可能掩盖不可绕过的高门槛，因此必须应用以下规则。

## 7.1 单点门槛下限

若存在不可绕过的核心门槛 \(M\)，则公共最终分通常不低于：

\[
M-0.5
\]

适用条件：

- 没有合理替代解法；
- 该门槛不只是实现细节；
- 不通过该门槛基本无法推进。

目标人群是否应用取决于其是否已掌握该门槛。

---

## 7.2 多个独立门槛

如果存在多个彼此独立的难点，不应只取平均。

定义高门槛为基础维度中：

\[
s_i\ge6.0
\]

且难点来源彼此独立。

| 独立高门槛数量 | 组合修正 |
|---:|---:|
| 0~1 | 0 |
| 2 | \(+0.2\sim+0.4\) |
| 3 | 累计 \(+0.5\sim+0.8\) |
| 4 或更多 | 根据具体结构说明，通常 \(+0.8\sim+1.2\) |

禁止把同一个观察拆成“建模、结论、转化”三个门槛重复计算。

---

## 7.3 短代码观察题约束

若同时满足：

- 前置知识门槛低；
- 算法设计简单；
- 实现短；
- 边界少；
- 核心规律可通过自然小数据发现；
- 错路不强；

则公共最终分通常不应超过：

\[
4.5
\]

若观察极其明显，可限制在：

\[
3.5\sim4.0
\]

如果结论难以自然发现，则不适用该上限。

---

## 7.4 实现题下限

若主思路常规但实现复杂度高：

- 实现分 `>= 7.0`：公共最终分通常不低于 `5.5`；
- 实现分 `>= 8.0` 且边界密集：公共最终分通常不低于 `6.0`。

---

# 8. 环境修正

v4 只允许对基础结构之外的赛时环境进行修正。

最终分：

\[
F=\operatorname{clamp}(B+C,0,10)
\]

其中 \(C\) 为环境修正总和。

普通题的环境修正建议控制在：

\[
-1.0\sim+1.0
\]

超过此范围必须说明其特殊性。

---

## 8.1 约束提示强度

| 情况 | 修正 |
|---|---:|
| 约束强烈提示唯一复杂度方向 | \(-0.3\sim-0.6\) |
| 有一定提示 | \(-0.1\sim-0.3\) |
| 无明显提示 | 0 |
| 约束具有迷惑性 | \(+0.2\sim+0.5\) |

---

## 8.2 官方样例提示

| 情况 | 修正 |
|---|---:|
| 样例直接暴露关键规律 | \(-0.2\sim-0.5\) |
| 覆盖主要边界 | \(-0.1\sim-0.2\) |
| 普通 | 0 |
| 样例弱或误导 | \(+0.1\sim+0.4\) |

不能因为知道正解后能解释样例，就认定样例有强提示。

---

## 8.3 题面表达质量

| 情况 | 修正 |
|---|---:|
| 题面特别清晰，定义严谨 | \(-0.1\sim-0.2\) |
| 正常 | 0 |
| 翻译差、定义歧义、示例不足 | \(+0.2\sim+0.6\) |

---

## 8.4 数据、pretest 与 hack 风险

必须区分：

- 理论正解难度；
- 赛时通过难度。

| 情况 | 修正 |
|---|---:|
| 数据明显弱，假做法可过 | 赛时通过难度下调 |
| pretest 弱、system test 强 | 不下调理论难度，说明赛时风险 |
| 数据强，卡边界充分 | 实现风险可小幅上调 |
| 数据未知 | 不修正 |

---

## 8.5 提交形式

| 类型 | 影响 |
|---|---|
| 普通在线编程 | 基准 |
| 结果填空 | 可允许慢程序与多重校验，通常下调实现和复杂度压力 |
| 输出答案 | 强调构造与验证，可能大幅改变难度 |
| 部分分 | 需要额外评价得分路径 |
| 交互 | 增加协议、信息获取和实现风险 |
| 通信 | 通常显著提高建模与实现负担 |
| 开放数据 | 需要考虑数据分析和特化策略 |

提交形式既可影响基础维度，也可产生环境修正。

---

## 8.6 时间、内存与语言限制

| 情况 | 修正 |
|---|---:|
| 时空非常宽松 | \(-0.1\sim-0.3\) |
| 正常 | 0 |
| 明显卡常或卡内存 | \(+0.2\sim+0.6\) |
| 特定语言显著不利 | 单独说明 |

如果性能优化本身是算法主体，应计入基础维度，而不是只作为修正。

---

# 9. 目标人群体感计算

目标人群体感必须通过七个维度重新评分。

例如一个树上博弈题：

- 公共前置知识门槛可能为 7.0；
- 若目标人群熟悉 SG，则目标前置知识门槛可降到 2.0；
- 若其实现能力一般，实现分不一定下降。

最终体感：

\[
F_{\text{target}}=\operatorname{clamp}\left(\sum_i w_i t_i+C_{\text{target}},0,10\right)
\]

rating gap 仅作检查：

```text
若目标人群 rating 远高于公共估分，但体感仍很高，
应检查是否存在冷门知识、长实现、几何、交互或专项短板。
```

禁止：

```text
逐项评分后，再按照 gap 机械减分。
```

# 10. CF 估分映射

v3 的线性公式：

\[
800+200F
\]

最大只能得到 2800，无法覆盖 2900～3500+，因此 v4 改为分段锚点映射。

## 10.1 锚点

| 最终分 \(F\) | CF 等价 rating |
|---:|---:|
| 0.0 | 800 |
| 1.0 | 1000 |
| 2.0 | 1200 |
| 3.0 | 1400 |
| 4.0 | 1600 |
| 5.0 | 1800 |
| 6.0 | 2000 |
| 7.0 | 2300 |
| 8.0 | 2600 |
| 9.0 | 3000 |
| 10.0 | 3500+ |

相邻锚点之间线性插值。

例如：

- \(F=4.5\Rightarrow1700\)
- \(F=6.5\Rightarrow2150\)
- \(F=7.5\Rightarrow2450\)
- \(F=8.5\Rightarrow2800\)
- \(F=9.5\Rightarrow3250\)

该映射是初始经验模型，后续应通过历史锚点题校准。

---

## 10.2 档位化

合法档位：

```text
800, 900, 1000, ..., 3500, 3500+
```

规则：

1. 连续估分 `< 850`：800；
2. `[850,3450)`：四舍五入到最近整百；
3. `[3450,3550)`：3500；
4. `>=3550`：3500+。

---

# 11. 赛时波动

赛时波动描述不同选手和不同场次之间体感的离散程度，不直接进入总分。

| 等级 | 特征 |
|---|---|
| 低 | 难度主要来自稳定可训练内容，选手表现集中 |
| 中 | 有一定观察或知识命中差异 |
| 高 | 单点观察、专题知识或路径狭窄导致明显分化 |
| 极高 | 想到即秒、想不到坐牢，且错误反馈很弱 |

波动来源可包括：

- 专题知识命中；
- 单点结论；
- 正解路径宽度；
- 错路反馈；
- 样例提示；
- 实现稳定性；
- 数据强弱。

重要规则：

```text
波动不加分。
导致波动的客观结构，已经分别计入基础维度。
```

---

# 12. 估分区间与置信度

## 12.1 波动对区间宽度的影响

| 赛时波动 | rating 区间宽度 |
|---|---|
| 低 | 中心值上下约 100 |
| 中 | 上下约 100~200 |
| 高 | 上下约 200~300 |
| 极高 | 上下约 300 或更大 |

区间端点必须落在合法 CF 档位。

---

## 12.2 估分置信度

| 置信度 | 条件 |
|---|---|
| 高 | 题面完整、正解明确、题型常见、数据环境已知 |
| 中 | 有一定观察或知识波动，但总体结构清晰 |
| 低 | 冷门专题、可能存在更简单解、数据未知或题面不完整 |

需要降低置信度的情况：

- 只看到一种复杂解法；
- 不确定是否存在民间简单解；
- 数据强弱未知；
- 官方题面缺失；
- 冷门 OJ 或特殊赛制；
- 题目高度依赖平台生态。


# 14. 主导难点与训练价值

## 14.1 主导难点

必须指出一到三个真正主导难度的因素，例如：

- 核心观察；
- 算法设计；
- 前置知识；
- 复杂度优化；
- 正确性证明；
- 实现工程；
- 错路排除。

不能机械重复所有维度。

## 14.2 训练价值

说明适合训练什么，例如：

- 状态设计；
- 复杂度降维；
- 构造；
- 贪心交换；
- 反例构造；
- 数据结构维护信息设计；
- 树形 DP；
- 博弈建模；
- 自动机计数；
- 性能工程；
- 题意形式化。

# 15. 输出格式与总评视觉规则

## 15.1 总评视觉规则

完整分析结束后，必须以独立的“总评”部分收束。

总评的目标是让读者不需要翻找正文，就能立即看到最重要的信息。

必须遵守：

1. **公共难度分和 CF 估分必须使用独立的 LaTeX 方框展示。**
2. **合理区间、目标人群体感、赛时波动和估分置信度集中放入一张紧凑表格。**
3. 总评中不展开计算公式，不重复评分过程。
4. 最后补充“题目定位”和“一句话评价”，每项不超过两句话。
5. 不要在总评后继续添加新的分析章节。
6. 即使正文较长，总评也应控制在一屏内。

推荐视觉结构：

```markdown
# 总评

\[
\boxed{\text{公共难度：}\ x.x/10.0}
\]

\[
\boxed{\text{CF 估分：}\ xxxx}
\]

| 目标人群体感 | 合理区间 | 赛时波动 | 估分置信度 |
|:---:|:---:|:---:|:---:|
| **x.x / 10.0** | **xxxx～xxxx** | **高** | **中** |

> **题目定位：** 偏难 Div.2 D / 常规 Div.2 E / 某专题入门题等。
>
> **一句话评价：** 用一句话概括核心难点、题型质量和总体难度。
```

其中：

- “公共难度”和“CF 估分”是最高优先级信息，必须各自使用独立方框；
- “目标人群体感”不能替代公共难度；
- rating 区间端点必须使用合法 CF 档位；
- 若估分置信度低，应在一句话评价中简要说明原因。

# 16. 评分流程

严格按以下顺序：

1. 阅读完整题面、约束、提交形式和样例；
2. 确认赛时合理可想到的最简单正解；
3. 写出发现路径，进行防后见之明检查；
4. 分别评价七个公共维度；
5. 根据目标画像分别评价七个目标维度；
6. 检查维度是否重复计费；
7. 应用单点门槛和独立多门槛规则；
8. 只对外部赛时环境进行修正；
9. 得到公共最终分和目标体感；
10. 使用分段锚点映射 CF rating；
11. 根据赛时波动确定合理区间；
12. 给出估分置信度；
13. 最后才与官方 rating 对照；
14. 使用固定总评卡片收束全文，且总评后不再追加分析。

# 17. 模型校准规则

单题评分时不得用官方 rating 反推答案。

但模型开发阶段可以使用历史题进行离线校准。

推荐流程：

1. 每个 200 分段选择若干锚点题；
2. 覆盖数学、DP、图论、数据结构、字符串、构造、实现、交互；
3. 多名评价者独立盲评；
4. 记录维度分、最终分、官方 rating、通过率、解题时间；
5. 统计平均绝对误差、中位绝对误差、题型偏差和评价者间方差；
6. 调整权重和映射锚点；
7. 不对单题强行拟合官方 rating。

---

# 18. 注意事项

1. 不使用当前题目的官方 rating 作为评分依据。
2. 官方 rating 只能用于事后对照或模型离线校准。
3. 所有基础维度固定纳入，不存在则记 0。
4. 公共难度与目标人群体感必须同时给出。
5. 目标人群按维度重新评分，不得再用 rating gap 二次校准。
6. 赛时波动不直接参与总分。
7. rating 区间应随波动变化。
8. 必须输出估分置信度。
9. 核心观察、形式化和算法设计不得重复计费。
10. 复杂度优化必须单独评价。
11. 猜到但难以确认正确的题，应提高“正确性论证”分。
12. 冷门知识应提高“前置知识门槛”，即使代码很短。
13. 结果填空、交互、通信和输出答案必须考虑提交形式。
14. 卡常、压内存和语言差异必须考虑性能工程。
15. 应按赛时合理可想到的最简单正解评分。
16. 若只知道复杂官方解而不确定是否有简单解，应降低置信度。
17. 发现规律时必须进行防后见之明检查。
18. 多个高门槛只有彼此独立时才应用组合修正。
19. 理论正解难度与赛时通过难度应在数据特殊时分开说明。
20. CF 估分必须落在合法档位：800、900、…、3500、3500+。
